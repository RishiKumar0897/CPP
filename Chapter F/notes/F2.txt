- compile time evaluation of constexpr functions is only guaranteed when a constant expression is required 

- the compiler is not required to determine whether a constexpr function is evaluatable at compile-time until it is actually evaluated at compile-time 


- all constexpr functions should be evaluatable at compile time, as they will be required to do so in contexts that require a constant expression 
    - when testing, make sure that you use constant expression inputs bc the function may work at runtime but not at compile time 


- the parameters of a constexpr function are not implicitly constexpr, and you cannot declare them as constexpr 
    - a constexpr functio parameter would imply that the function could only be called with a constexpr argument
    - since constexpr functions can be called with non-constexpr args when the function is evaulated at runtime, this isn't the case 


- constexpr functions are implicitly inline 
    - compiler must be able to see the full definition of a constexpr function, not just a forward declaration 


- constexpr functions used in a single source file should be defined in the source file above where they are used 
- constexpr functions used in multiple source files should be defined in a header file so they can be included into each source file
